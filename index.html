// 01 — Problem → Solution

業界の構造的課題
と技術的解決

ISSUE_01
▲
☁
大手クラウドによるBAN・凍結リスク
AWS・Cloudflareはアダルトコンテンツを利用規約で制限。予告なしのアカウント凍結はサービス停止に直結し、事業継続の最大リスクになる。
// Solution
  • FlokiNET / Bunny.net など明示許可サービスを選定
  • Docker Composeで環境をコードとして管理・移行可能に
  • DNS TTLを300秒以下に維持し切替時間を最小化
  • Runbook整備で30分以内の復旧を保証
ISSUE_02
▼
⚡
WordPressの肥大化と表示速度劣化
画像・動画リッチなサイトはWordPressのデータ蓄積で年々重くなる。表示速度の劣化は直接的な機会損失であり、既存スタッフの操作習熟度も捨てられない。
// Solution
  • Next.jsでヘッドレス化、WPは管理画面のみに限定
  • ISR（revalidate=60）でコンテンツ更新を60秒以内に反映
  • 画像をWebP/AVIF自動変換しBunny CDNでエッジ配信
  • 管理画面はWordPressのまま維持し教育コストをゼロに抑える
ISSUE_03
▼
🔒
キャスト投稿画像の個人情報漏洩リスク
スマートフォンで撮影した画像にはGPS座標・機種名・撮影日時が埋め込まれている。そのまま公開すれば住所特定に直結し、キャストの安全が損なわれる。
// Solution
  • サーバサイドパイプラインでEXIF除去を全投稿に強制適用
  • sharpでメタデータを空にしてからBunny.netに保存
  • ffmpegで動画メタデータも同時に除去・HLS変換
ISSUE_04
▼
📊
業務管理のアナログ依存・内製化の遅れ
シフト管理・メディア投稿・キャスト情報の更新がバラバラのツールに分散。LINEでの連絡漏れ・Excelでの管理ミスが業務コストを押し上げている。
// Solution
  • JWT認証付きシフト管理APIで申請・承認フローを内製化
  • Telegram Botで承認通知を自動化し連絡漏れをゼロに
  • 経営ダッシュボードで全店舗の状況をリアルタイム俯瞰

// 04 — Philosophy

行動指針

PRINCIPLE_01
「レガシーを否定しない」
WordPressは世界シェア40%超のCMS。現場スタッフの操作習熟度という「人的資産」も含めたシステムだ。フロントのヘッドレス化で技術的負債を解消しつつ、管理画面はWordPressを維持することで教育コストをゼロに抑える。
PRINCIPLE_02
「技術を目的化しない」
最新技術の採用は、ビジネス課題の解決に貢献する場合にのみ正当化される。複数店舗規模にマイクロサービスはオーバースペック。Docker Composeによる単一ホスト構成でポータビリティを確保するほうが合理的だ。
PRINCIPLE_03
「法は越えない。
グレーはいとわない」
風営法・条例を遵守した上で、大手クラウドの利用規約の外側にある合法的なインフラ構成を活用する。法的に黒でないことと、大手プラットフォームのポリシー外であることは、まったく別の話だ。
